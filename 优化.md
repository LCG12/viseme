但它目前最主要的问题不是“8个 viseme 太少”，而是下面三点：

1. **RMS只能判断有没有声音，不能准确判断音素边界；**
2. **每个音节只做一次首嘴型到尾嘴型的插值，协同发音太简单；**
3. **直接在舵机角度空间线性插值，不一定对应自然的嘴部形变。**

建议按以下顺序优化。

---

# 一、最高优先级：不要用 RMS 分配音节边界

你现在的方案是：

1. 用 RMS 找到整段有效语音；
2. 把有效帧分配给若干音节；
3. 每个音节内部插值。

这个方法对音量比较均匀的 TTS 可以工作，但存在明显问题。

## 1. RMS看不到无声辅音和闭唇准备

例如：

- `b、p、m` 的闭唇动作可能在声音出现前就开始；
- `f、s、sh` 的能量可能较弱；
- 爆破音前存在短暂闭塞；
- 一个元音往往比辅音能量更强。

因此，RMS高的地方不一定是一个完整音节，也不能准确区分：

b+a

中的闭唇阶段和张嘴阶段。

例如“ba”可能被模型处理为整段均匀地：

V0​→V7​

但真实过程更接近：

提前闭唇→短暂保持→快速释放→保持元音嘴型

## 2. 更好的方式：使用音素级时间对齐

将流程改成：

音频+文本→音素时间戳→viseme轨迹

每个音素都应该获得：

```
phonestart_timeend_timeviseme_id
```

例如“爸爸”的对齐结果可以是：

| 音素  | 起始时间   | 结束时间   | Viseme |
| --- | ------ | ------ | ------ |
| b   | 120 ms | 175 ms | V0     |
| a   | 175 ms | 390 ms | V7     |
| b   | 390 ms | 445 ms | V0     |
| a   | 445 ms | 680 ms | V7     |

如果使用 TTS，优先直接读取 TTS 引擎生成的音素或字词边界。如果没有时间戳，再使用强制对齐工具。

RMS仍然有用，但用途应该改为：

- 裁剪首尾静音；
- 检测整体发声强度；
- 调整张嘴幅度；
- 辅助检测停顿。

不要再让 RMS 承担音素切分的主要任务。

---

# 二、将“音节首尾插值”改成“音素事件序列”

目前一个音节只有：

```
start_visemeend_visemealpha
```

例如：

```
ba: V0 → V7
```

这会造成所有音节都采用几乎相同的运动模板，只是起止嘴型不同。

建议把每个音节进一步展开成音素事件：

```
{  "syllable": "ba",  "phones": [    {      "phone": "b",      "viseme": 0,      "start_ms": 120,      "end_ms": 175,      "dominance": 1.0    },    {      "phone": "a",      "viseme": 7,      "start_ms": 175,      "end_ms": 390,      "dominance": 0.8    }  ]}
```

这里的 `dominance` 表示该音素的嘴型约束强度。

例如：

- `b、p、m` 必须闭唇，支配强度高；
- `f` 必须形成唇齿关系，支配强度高；
- 元音持续时间长，但可以受到前后音素影响；
- 舌位不同、但机器人看不出来的辅音，嘴型支配强度可以较低。

---

# 三、加入协同发音，而不是逐音节独立运动

人说话时，嘴型会提前准备下一个音。例如说：

```
我爱你wo ai ni
```

在 `wo` 结束前，嘴可能已经开始从圆唇向 `ai` 的大开口过渡。

你现在的独立音节插值容易在边界处出现：

- 先到达当前音节的完整终点；
- 然后突然开始下一音节；
- 轨迹有明显分段感。

## 推荐方法：重叠的 viseme 激活曲线

对于每一个音素 i，定义一个随时间变化的权重：

wi​(t)

最终嘴型不是只在两个 viseme 之间插值，而是多个邻近音素共同作用：

P(t)=∑i​di​wi​(t)∑i​di​wi​(t)Pi​​

其中：

- Pi​：第 i 个音素对应的静态嘴型；
- wi​(t)：它在当前时刻的激活程度；
- di​：音素的支配权重。

例如闭唇音可以提前约40～80 ms开始生效：

```
实际发音时刻：500 ms闭唇准备开始：430 ms
```

元音则可以在音素结束后保留少量时间。

这样可以自然形成：

前一个音素+当前音素+下一个音素

的协同发音。

---

# 四、不要只使用线性 alpha

你已经提出将线性插值改成：

3α2−2α3

这是正确方向，但建议根据动作类型使用不同曲线。

## 1. 普通嘴型过渡

使用 smootherstep：

αs​=6α5−15α4+10α3

它的起点和终点速度、加速度都更平滑。

## 2. 闭唇音

`b、p、m` 应该使用：

- 快速进入闭唇；
- 短暂保持；
- 快速释放。

不能只是平滑地经过闭唇状态。

例如将一个音素分成：

```
进入：25%保持：45%释放：30%
```

## 3. 元音

元音更适合：

- 较快进入；
- 较长保持；
- 较慢离开。

例如：

```
进入：25%保持：55%离开：20%
```

所以需要为 viseme 增加运动类型：

```
viseme_id,motion_profile,attack_ratio,hold_ratio,release_ratio0,closure,0.25,0.45,0.307,vowel_open,0.25,0.55,0.20
```

---

# 五、不要直接在线性舵机角度空间混合

你目前的计算是：

q(t)=(1−α)qstart​+αqend​

其中 q 是6个舵机角度。

这虽然简单，但机器人皮肤和拉索通常是非线性的：

- 舵机转动10度，皮肤不一定移动10%的总行程；
- 拉索前半段可能在消除松弛；
- 左右嘴角会相互拉扯；
- 张嘴时，上下嘴唇的作用会变化；
- 同时激活两个姿态可能导致过度拉伸。

论文通过 LBS、运动学模拟器和逆运动学解决这类问题，而不是简单假设执行器空间完全线性。

## 第一阶段可采用分段曲线

为每个舵机标定：

归一化动作强度→实际舵机角度

例如：

| 强度   | mouth_open角度 |
| ---- | ------------ |
| 0.0  | 90           |
| 0.25 | 98           |
| 0.5  | 111          |
| 0.75 | 129          |
| 1.0  | 152          |

然后使用查表插值，而不是：

```
0.5强度 = 中性角度与最大角度简单取平均
```

每个舵机都建立独立的响应曲线：

```
servo_response/mouth_open.csvservo_response/corner_l.csv...
```

## 第二阶段加入组合补偿

例如张嘴越大，嘴角能够安全上提的范围越小：

qcorner​=qcornerraw​−k⋅ujaw​ucorner​

上嘴唇和嘴巴开合也可能需要耦合补偿：

qupper​=qupperraw​+k1​ujaw​+k2​ujaw​uupper​

可以先人工拟合少量交互项，不必立即建立复杂神经网络。

---

# 六、静态 viseme 不要只“人工选最好的一次”

你当前计划是每个 viseme 调3～5次，然后人工挑选最自然的一组。

这种方式的问题是：

- 不同人选择标准不一致；
- 最自然的单独嘴型不一定最适合连续运动；
- 静态看起来好，和其他嘴型混合时可能很差。

建议每个 viseme 保留一个**候选区域**，而不是只保存一个姿态。

例如：

```
V7_a_smallV7_a_mediumV7_a_large
```

然后根据音频强度决定开口幅度：

PV7​(t)=Psmall​+e(t)(Plarge​−Psmall​)

其中 e(t) 是归一化能量。

这样“a”的嘴型不再永远只有一个固定张嘴幅度。

## 建议每个元音至少采集三档

| Viseme  | 弱    | 正常   | 强    |
| ------- | ---- | ---- | ---- |
| V3 圆唇   | 小圆唇  | 正常圆唇 | 强圆唇  |
| V5 扁平   | 小幅展开 | 正常展开 | 大幅展开 |
| V6 圆形开口 | 小    | 中    | 大    |
| V7 大开口  | 小    | 中    | 大    |

闭唇类 V0 通常不需要多档，但需要保证真正闭合。

---

# 七、用 RMS 控制幅度，而不是控制时长

这是对你现有 RMS 模块最自然的改造。

现在：

RMS→分配帧数

建议改成：

音素对齐→时间位置 RMS→动作幅度

例如：

a(t)=clip(rmax​−rmin​RMS(t)−rmin​​,0,1)

然后：

umouthOpen​(t)=uphoneme​(t)(0.6+0.4a(t))

也就是：

- 发音决定嘴型类别；
- 能量决定动作幅度；
- 不让低能量直接变成闭嘴；
- 不让高能量改变音素类别。

RMS最好先进行：

- 对数变换；
- 滑动平均；
- 百分位归一化；
- 峰值限制。

不要用每条音频的最大值直接归一化，否则某一个爆音会压低整段动作幅度。

---

# 八、8类 viseme 可以保留，但应重新判断哪些需要拆分

你们机器人的嘴部自由度有限，没有舌头控制，所以不需要把所有音素都拆得很细。

真正值得区分的是视觉上能够被机器人表达的类别。

建议保留或调整为：

| 类别  | 主要声音     | 机器人表现重点      |
| --- | -------- | ------------ |
| V0  | b、p、m    | 完全闭唇         |
| V1  | f        | 上齿/下唇接近的近似动作 |
| V2  | d、t、n、l  | 小开口、中性嘴角     |
| V3  | w、u      | 小开口、收圆       |
| V4  | e、en、eng | 中等开口         |
| V5  | i、j、q、x  | 扁平、横向展开但不微笑  |
| V6  | o、ong    | 圆形中大开口       |
| V7  | a、an、ang | 大开口          |

但需要特别检查：

## `zh、ch、sh、r`

这些音的唇形通常和 `d、t、n、l` 不完全相同，不过你们没有舌头自由度，单独拆分类别的收益可能有限。

## `z、c、s`

放在 V5 可能容易让机器人产生“微笑感”。可以让它们使用较弱的横向展开。

因此，可以允许同一个 viseme 带不同的幅度修正：

```
i  → V5，spread_scale=1.0s  → V5，spread_scale=0.45x  → V5，spread_scale=0.65
```

而不是要求同类音素完全使用相同舵机姿态。

---

# 九、重点优化你们六个嘴部通道的职责

你们的嘴部执行器是：

```
8  mouth_open9  mouth_up_corner_l10 mouth_up_corner_r11 mouth_lip_upper_l12 mouth_lip_upper_r13 mouth_lip_lower
```

建议不要让所有通道都直接跟随同一个 viseme 插值。

## mouth_open

主要表达：

- 元音开口大小；
- 下颌开合；
- 音量幅度。

## mouth_up_corner_l/r

主要表达：

- 横向展开；
- 左右不对称；
- 微笑或情绪。

对于纯语音嘴型，它们的幅度要小。特别是 `i`，不要通过大幅上提嘴角实现，否则会一直像在笑。

## mouth_lip_upper_l/r

主要表达：

- 上唇抬起；
- 闭唇释放；
- 左右细微不对称；
- `f` 等唇齿音的近似形状。

## mouth_lip_lower

主要表达：

- 下唇下降；
- 闭唇；
- 元音开口修正；
- `f` 的下唇位置。

## cheek_l/r

暂时不要作为主要语音嘴型通道。

脸颊更适合表达：

- 微笑；
- 情绪；
- 说话时的低频表情变化。

如果脸颊逐音素快速运动，容易产生不自然的抖动。

# 十、建立单独的“发音层”和“表情层”

你们最终有16个执行器，不应把所有动作都混在 viseme 系统里。

建议输出拆成：

ut​=utarticulation​+utexpression​+utbehavior​

其中：

## 发音层

控制：

```
mouth_openupper lipslower lip少量 mouth corners
```

## 表情层

控制：

```
mouth cornerscheeksbrows
```

例如机器人处于开心状态时：

ucorner​=uspeech​+0.3uhappy​

但需要限制总和，避免嘴角过拉。

## 行为层

控制：

```
blinkgazeattention
```

眼球方向不应该来自 viseme。

# 十一、补偿舵机延迟，而不只是添加全局 offset

你当前准备使用统一的：

```
servo_audio_offset_ms
```

这可以解决整体偏早或偏晚，但不同舵机的响应时间可能不同：

- `mouth_open` 可能较快；
- 上嘴唇拉索可能较慢；
- 左右嘴角摩擦不同；
- 回程速度和拉动速度可能不一致。

建议对每个舵机做阶跃测试。

例如从中性发送到50%动作，测量：

- 开始移动延迟；
- 到达50%的时间；
- 到达90%的时间；
- 回程时间。

保存：

```
servo_id,onset_delay_ms,rise_time_ms,fall_time_ms,deadband8,35,110,135,0.049,50,150,170,0.07
```

然后分别进行时间补偿：

qicmd​(t)=qitarget​(t+Δi​)

不要所有舵机都使用同一个时间偏移。

---

# 十二、控制频率可以分成两层

模型或 viseme 轨迹仍然可以保持25 Hz：

40 ms一帧

但发送给舵机前，可以插值到：

50 Hz或100 Hz

流程变成：

```
25 Hz viseme目标→ 平滑插值→ 100 Hz执行器指令
```

这样可以减少舵机每40 ms一次明显跳变。

需要注意，不能只是重复同一帧，而应在相邻目标之间做连续插值，并执行速度和加速度限制。

---

# 十三、使用音频时间戳驱动，而不是 `sleep(0.04)`

实时执行时，最容易出现的问题是时间逐渐漂移。

不建议：

```
for frame in frames:    send(frame)    time.sleep(0.04)
```

因为发送、计算和系统调度也会花时间，最后机器人会逐渐落后音频。

应使用音频时钟作为主时钟：

```
target_frame = int(audio_playback_time * 25)
```

每次根据当前音频播放位置选择对应轨迹帧。

执行日志应同时保存：

```
planned_timeaudio_timecommand_send_timeactual_feedback_time
```

这样才能判断问题来自：

- 嘴型规划；
- 程序发送；
- 通信；
- 舵机机械响应。

# 十四、评价体系需要增加客观指标

目前的人工指标：

- lip sync；
- clarity；
- naturalness。

这些很重要，但不足以定位问题。

建议增加以下客观指标。

## 1. 闭唇成功率

对于 `b、p、m`：

closure rate=应闭唇次数正确闭唇次数​

## 2. 开口峰值时间误差

比较：

- 音频元音能量峰值；
- `mouth_open` 轨迹峰值。

得到：

Δtpeak​

## 3. 指令饱和率

统计舵机有多少帧达到最大或最小限制：

saturation rate=总帧数限位帧数​

饱和率高说明 viseme 姿态或混合权重过激。

## 4. 轨迹平滑度

计算速度、加速度或 jerk：

J=t∑​∥qt​−3qt−1​+3qt−2​−qt−3​∥2

数值太高意味着动作生硬或抖动。

## 5. 左右对称误差

对对称发音检查：

∣qleft​−qright​∣

避免机械标定差异造成嘴型歪斜。

# 十五、优化数据文件结构

你现在为每句话生成多个 CSV 和 JSON，调试很清晰，但句子多以后会变得难以管理。

建议保留一个总 manifest：

```
sentence_id,text,audio_file,duration,split,speaker
```

每条语音使用一个统一的 NPZ 或 Parquet 文件保存：

```
timestampsphonesviseme_weightsrmsservo_targetsservo_actual
```

例如：

```
np.savez(    "S001.npz",    timestamps=timestamps,    phone_ids=phone_ids,    viseme_weights=weights,    rms=rms,    servo_targets=targets,    servo_actual=actual,)
```

CSV可以保留作为人工检查版本，但不要让运行程序依赖大量零散CSV。

# 十六、后续可以升级成小型学习模型

等规则系统稳定后，可以用已经采集的数据训练一个很小的时序模型。

输入：

音素类别+前后音素+音素内位置+RMS+上一帧舵机状态

输出：

6个嘴部舵机目标

例如：

xt​=[phonet−4:t+4​,RMSt​,progresst​,qt−1​] qt​=f(xt​)

可采用：

- 一维卷积；
- GRU；
- 小型 Transformer；
- 论文中的时间融合模块。

这时不需要完整复制论文的5023个人脸顶点和51个 Blendshape。你们可以直接学习：

音素上下文→6维机器人嘴部控制

但前提是先拥有质量较高的目标轨迹。

1. 规则系统生成轨迹；
2. 人工修改不自然部分；
3. 保存修改后的轨迹；
4. 用这些轨迹训练模型；
5. 模型学习协同发音和舵机动态。

# 十七、建议采用三轮优化路线

## 第一轮：不训练模型

重点改动：

1. RMS只用于裁剪和幅度控制；
2. 使用音素级时间戳；
3. 加入闭唇保持和元音保持；
4. 将线性插值换成分段平滑曲线；
5. 分舵机标定延迟；
6. 25 Hz轨迹插值到更高控制频率。

这一轮通常能得到最大提升。

## 第二轮：加入协同发音

重点改动：

1. 当前音素、前一个音素和后一个音素共同决定嘴型；
2. 闭唇音提前准备；
3. 元音动作延后释放；
4. 为不同音素设置 dominance；
5. 增加组合补偿和舵机非线性曲线。

## 第三轮：数据驱动优化

重点改动：

1. 使用摄像头拍摄机器人嘴部；
2. 自动提取嘴角、上下唇和开口高度；
3. 对照目标嘴型优化舵机姿态；
4. 收集人工修正轨迹；
5. 训练小型时序网络输出6维舵机控制。
